# CLAUDE.md — Project orientation

Repo-specific terms, layout, and conventions for Claude / agents.  Keep this
short; deeper architecture lives in `docs/architecture.md`.

## Repo layout

```
appunvs/                    ← host shell (iOS + Android user-facing app)
  ios/Runtime/                Swift app: Chat / Stage / Profile tabs
  android/app/.../runtime/    Kotlin app, same three tabs
relay/                      ← Go service (auth, AI agent, build, artifact, /ws sync)
  internal/box/               box service + Events pub/sub
  internal/sandbox/           Builder interface + LocalStub + DockerBuilder
  internal/handler/           HTTP handlers (incl. /box/events SSE)
runtime/                    ← shared RN project AND SDK source
  src/                        TypeScript: HostBridge + fixture/test components
  ios/SDK/                    ObjC++ framework sources → RuntimeSDK.xcframework
  android/runtimesdk/         Kotlin library → runtime.aar
  sandbox/                    Dockerfile + metro.config.js for AI-bundle build
                              + fixture-rn/ (the test bundle's TS source)
  packaging/                  build-ios.sh / build-android.sh / build-fixture.sh
                              / build-sandbox.sh
shared/                     ← protobuf + cross-platform schemas
docs/                       ← architecture / auth / protocol / providers / conventions
```

## Glossary

Project-specific terms used throughout the codebase:

- **AI bundle** — a JS bundle generated by AI editing code in a Box, then
  built by sandbox + metro into a runnable artifact.  Loaded into a
  `RuntimeView` on the host.

- **Box** — a remote workspace on the relay holding one project's git
  repo + its compiled bundles.  AI agents edit files inside it via tools
  (`fs_write`, `publish_box`).  Identity = `{boxID, version, title}`.

- **Bundle** — Hermes-ready JS the host downloads and runs.  Two flavors:
  AI-generated (per-Box) and fixture (test-only).

- **Fixture** — minimal scaffolding whose only purpose is to make a test
  runnable.  E.g. **fixture bundle** = `runtime/sandbox/fixture-rn/index.tsx`
  compiled to `RuntimeRoot.jsbundle`, used by D3.c.4 UI tests to verify
  RuntimeView loads + renders.  E.g. **fixture app** = a tiny iOS/Android
  app whose only job is to host a `RuntimeView` so an XCUITest /
  androidTest can assert against the rendered output.

- **Host shell / host app** — the user-facing app that lives in
  `appunvs/{ios,android}/`.  Embeds the runtime SDK, provides Chat / Stage
  / Profile tabs, owns auth + box-events SSE consumer.

- **Relay** — Go service in `relay/` mediating: auth, AI turns, sandbox
  builds, artifact storage, namespace sync (`/ws`), and box-events fanout
  (`/box/events`).

- **RuntimeSDK** — the framework / library shipped from `runtime/` that
  host apps embed.  iOS: `RuntimeSDK.xcframework`.  Android: `runtime.aar`.
  Public surface: `RuntimeView`, `RuntimeBoxIdentity`, `AppunvsHostModule`.

- **RuntimeView** — UI component (UIView on iOS, FrameLayout on Android)
  that mounts an AI bundle in its own per-instance Hermes runtime.
  `loadBundle(at:)` boots it; `reset()` tears it down.

- **Sandbox** — isolated build environment for turning AI source into a
  runnable bundle.  Two implementations of `sandbox.Builder` today:
  `LocalStub` (placeholder, doesn't really build) and `DockerBuilder`
  (real metro inside `appunvs/sandbox` docker image).  Future: ECI /
  Modal / Vercel Sandbox / Firecracker.

- **AI Engine** — `relay/internal/ai/Engine` interface that turns one
  Chat turn into a stream of Token / ToolCall / ToolResult / Finished
  frames.  Three implementations: `StubEngine` (echo, no provider hits;
  CI default), `OpenAIEngine` (any OpenAI-compatible endpoint —
  DeepSeek / Volcengine / Moonshot / Zhipu / Dashscope), and
  `AnthropicEngine` (direct Anthropic Messages API for Claude).
  Backend selection via `cfg.AI.Backend` = `stub` | `anthropic` |
  `<provider id>`.

- **SubRuntime** — one Hermes JS runtime, scoped to a single bundle load
  in a single RuntimeView.  Cross-bundle state cannot leak; reset =
  destroy.

- **Tier 1 / Tier 2 / Forbidden modules** — `runtime/MODULES.md`'s
  allowlist of npm packages AI bundles may import.  Enforced at metro
  bundle time by `runtime/sandbox/metro.config.js`.

- **Host bridge** — `runtime/src/HostBridge.ts`.  TypeScript surface AI
  bundles `import` from `@appunvs/host` to reach native capabilities
  (storage, network, publish, subscribe).  Implemented natively by
  `AppunvsHostModule` (`runtime/ios/SDK/AppunvsHostModule.mm` /
  `runtime/android/runtimesdk/.../AppunvsHostModule.kt`); host shells
  register handler closures via `RuntimeBridgeWiring`.

## iOS xcodeproj layout (non-obvious)

`runtime/ios/` has two xcodeprojs by design — APFS is case-insensitive on
macOS, so `RuntimeSDK.xcodeproj` and `RuntimeSdk.xcodeproj` would clobber
each other.  We name them differently:

- **`RuntimeSdk.xcodeproj`** (lowercase k) — RN init's dev-harness app,
  hand-edited.  Exists so `pod install` has something to integrate against.
- **`AppunvsRuntimeSDK.xcodeproj`** — generated by XcodeGen from
  `runtime/ios/SDK.yml`.  Contains the framework target (binary still
  ships as `RuntimeSDK.framework`) plus, post-D3.c.4, the fixture app +
  XCUITest targets.

`Podfile` integrates both.  The xcworkspace (`RuntimeSdk.xcworkspace`,
created by pod install) wraps both into one Xcode workspace.

## Build flow

| Artifact                           | Producer script                     | Used by                      |
|------------------------------------|-------------------------------------|------------------------------|
| `runtime/build/ios/RuntimeSDK.xcframework` | `runtime/packaging/build-ios.sh`     | host iOS app embeds          |
| `runtime/build/android/runtime.aar`        | `runtime/packaging/build-android.sh` | host Android app embeds      |
| `runtime/build/fixture/{ios,android}/RuntimeRoot.jsbundle` | `runtime/packaging/build-fixture.sh` | D3.c.4 UI tests              |
| `appunvs/sandbox:latest` docker image       | `runtime/packaging/build-sandbox.sh` | relay's `DockerBuilder`      |

## CI workflows

- `.github/workflows/ci.yml` — relay (Go), golangci-lint, sandbox docker
- `.github/workflows/native.yml` — host iOS / host Android / runtime SDK
  iOS / runtime SDK Android (Library) / runtime SDK Android (instrumented)
  / runtime fixture bundle
- `.github/workflows/lint.yml` — formatter / lint passes

When a CI job fails on a PR, the job's "Post failure log to PR" step
captures the last ~300 lines of relevant logs and posts them as a PR
comment — failures are visible without repo admin access to the actions
log download endpoint.
